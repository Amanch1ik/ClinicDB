Целью данной работы является разработка реляционной базы данных для автоматизации работы медицинской клиники. В ходе работы необходимо было решить следующие задачи:

- обеспечить целостность данных на уровне базы;

Клиника — довольно сложная предметная область, потому что здесь пересекается сразу несколько направлений: учёт пациентов, работа врачей, расписание, медицинская документация, анализы и финансы. Важно было не запутаться и грамотно разделить это на таблицы.

## 2. Анализ предметной области

Прежде чем проектировать таблицы, было важно понять, что вообще происходит в клинике и что нужно хранить.

Основные процессы:
1. **Регистрация пациента** — при первом обращении заводится карточка с контактными данными.
2. **Запись на приём** — пациент записывается к конкретному врачу на конкретное время.
3. **Проведение приёма** — врач ведёт приём, ставит диагноз, назначает лечение, при необходимости направляет на анализы.
4. **Лабораторные исследования** — результаты анализов привязываются к соответствующей медицинской карте.
5. **Оплата** — после приёма пациент оплачивает услуги.
6. **Доступ к системе** — разные сотрудники имеют разный уровень доступа: врач видит только своих пациентов, регистратор работает с расписанием, администратор управляет всем.

Из этого анализа были выделены 7 основных сущностей.

## 3. Выбор СУБД

Для реализации была выбрана **PostgreSQL 15**. Основные причины:

- полноценная поддержка реляционной модели и внешних ключей;
- наличие типа `ENUM` — удобно для статусов и ролей;
- тип `NUMERIC` для точного хранения денежных сумм (без погрешностей с плавающей точкой);
- бесплатная и широко используется на практике;
- хорошая документация.

## 4. Структура базы данных

### 4.1. ER-диаграмма (схема сущностей)

```
patients (Пациенты)
    │
    ├──< appointments (Приёмы) >──── doctors (Врачи)
    │                                    │
    ├──< medical_records (Медкарты) >────┘
    │       │
    │       └──< lab_tests (Анализы)
    │
    └──< payments (Платежи)

users (Пользователи) ──── doctors (необязательная связь)
```

Обозначения: `<` и `>` — сторона «многие» в связи один-ко-многим.

### 4.2. Таблицы и их описание

#### Таблица `patients` — Пациенты

Хранит основную информацию о пациентах клиники.

| Поле | Тип | Описание |
|------|-----|----------|
| id | SERIAL PK | Уникальный идентификатор |
| last_name | VARCHAR(60) NOT NULL | Фамилия |
| first_name | VARCHAR(60) NOT NULL | Имя |
| middle_name | VARCHAR(60) | Отчество (необязательно) |
| birth_date | DATE NOT NULL | Дата рождения |
| gender | ENUM('М','Ж') NOT NULL | Пол |
| phone | VARCHAR(20) UNIQUE | Номер телефона |
| email | VARCHAR(120) UNIQUE | Электронная почта |
| address | TEXT | Адрес проживания |
| created_at | TIMESTAMP DEFAULT NOW() | Дата регистрации в системе |

Поля `phone` и `email` помечены как `UNIQUE` — у одного пациента может быть только один номер телефона в системе, и по номеру можно однозначно найти пациента. Отчество сделано необязательным, так как у некоторых пациентов его нет.

#### Таблица `doctors` — Врачи

Информация о сотрудниках клиники, ведущих приёмы.

| Поле | Тип | Описание |
|------|-----|----------|
| id | SERIAL PK | Уникальный идентификатор |
| last_name | VARCHAR(60) NOT NULL | Фамилия |
| first_name | VARCHAR(60) NOT NULL | Имя |
| middle_name | VARCHAR(60) | Отчество |
| specialty | VARCHAR(100) NOT NULL | Специальность |
| phone | VARCHAR(20) UNIQUE | Рабочий телефон |
| email | VARCHAR(120) UNIQUE | Рабочий email |
| office | VARCHAR(20) | Номер кабинета |
| created_at | TIMESTAMP DEFAULT NOW() | Дата добавления в систему |

#### Таблица `users` — Пользователи системы

Хранит учётные данные для входа в информационную систему клиники. Это отдельная таблица — не стоит смешивать сотрудников и логины, потому что, например, администратор системы может вообще не быть врачом.

| Поле | Тип | Описание |
|------|-----|----------|
| id | SERIAL PK | Уникальный идентификатор |
| login | VARCHAR(60) NOT NULL UNIQUE | Логин для входа |
| password_hash | VARCHAR(255) NOT NULL | Хэш пароля (bcrypt) |
| role | ENUM NOT NULL | Роль: администратор / врач / регистратор |
| email | VARCHAR(120) NOT NULL UNIQUE | Email пользователя |
| doctor_id | INT FK → doctors | Ссылка на врача (только для роли «врач») |
| created_at | TIMESTAMP DEFAULT NOW() | Дата создания учётной записи |

Пароль хранится только в виде хэша (bcrypt). Хранить открытый пароль в базе — грубая ошибка безопасности, которую нельзя допускать даже в учебном проекте.

Поле `doctor_id` заполняется только для пользователей с ролью «врач» и позволяет системе понять, какой врач сейчас работает и чьих пациентов ему показывать.

Роли разграничивают доступ:
- **Администратор** — полный доступ ко всем данным;
- **Врач** — видит только своих пациентов и их медкарты;
- **Регистратор** — работает с расписанием и карточками пациентов, но без доступа к медицинским данным.

#### Таблица `appointments` — Расписание приёмов

Центральная таблица для управления расписанием.

| Поле | Тип | Описание |
|------|-----|----------|
| id | SERIAL PK | Уникальный идентификатор |
| patient_id | INT NOT NULL FK → patients | Пациент |
| doctor_id | INT NOT NULL FK → doctors | Врач |
| appointment_date | DATE NOT NULL | Дата приёма |
| appointment_time | TIME NOT NULL | Время приёма |
| type | ENUM NOT NULL | Тип: консультация / анализ / процедура |
| status | ENUM NOT NULL DEFAULT 'запланирован' | Статус: запланирован / проведен / отменен |
| notes | TEXT | Дополнительные заметки |
| created_at | TIMESTAMP DEFAULT NOW() | Когда запись была создана |

Важный момент — составное ограничение уникальности:

```sql
CONSTRAINT uq_doctor_slot UNIQUE (doctor_id, appointment_date, appointment_time)
```

Оно не позволяет записать двух разных пациентов к одному врачу в одно и то же время. Без этого ограничения при параллельных запросах могло бы возникнуть «двойное бронирование».

#### Таблица `medical_records` — Медицинские карты

Каждая запись соответствует одному визиту пациента. Здесь хранится диагноз и назначения врача.

| Поле | Тип | Описание |
|------|-----|----------|
| id | SERIAL PK | Уникальный идентификатор |
| patient_id | INT NOT NULL FK → patients | Пациент |
| doctor_id | INT NOT NULL FK → doctors | Врач, заполнивший карту |
| visit_date | DATE DEFAULT CURRENT_DATE | Дата визита |
| diagnosis | TEXT NOT NULL | Диагноз |
| prescription | TEXT | Назначения и рекомендации |
| notes | TEXT | Дополнительные примечания |
| created_at | TIMESTAMP DEFAULT NOW() | Дата создания записи |

Обратите внимание: для `doctor_id` здесь использовано `ON DELETE RESTRICT`, а не `CASCADE`. Это означает, что врача нельзя удалить из системы, пока за ним числятся медицинские карты. Это сделано намеренно — медицинская документация должна оставаться в системе даже если сотрудник уволился.

#### Таблица `lab_tests` — Лабораторные анализы

Результаты анализов всегда привязаны к конкретной медицинской карте.

| Поле | Тип | Описание |
|------|-----|----------|
| id | SERIAL PK | Уникальный идентификатор |
| medical_record_id | INT NOT NULL FK → medical_records | Медицинская карта |
| test_name | VARCHAR(150) NOT NULL | Название анализа |
| test_date | DATE NOT NULL | Дата проведения |
| result | TEXT | Результат |
| notes | TEXT | Примечания |
| created_at | TIMESTAMP DEFAULT NOW() | Дата добавления в систему |

Одна медицинская карта может содержать несколько анализов. Например, при кардиологическом приёме врач может назначить и ЭКГ, и анализ крови — оба результата попадут в одну медкарту.

#### Таблица `payments` — Финансовые операции

Учёт оплаты услуг клиники.

| Поле | Тип | Описание |
|------|-----|----------|
| id | SERIAL PK | Уникальный идентификатор |
| patient_id | INT NOT NULL FK → patients | Пациент |
| amount | NUMERIC(10,2) NOT NULL CHECK > 0 | Сумма |
| payment_date | DATE DEFAULT CURRENT_DATE | Дата платежа |
| type | ENUM NOT NULL | Тип: оплата приёма / оплата процедуры |
| method | ENUM NOT NULL | Способ: наличные / карта / онлайн |
| status | ENUM DEFAULT 'не оплачено' | Статус: оплачено / не оплачено |
| created_at | TIMESTAMP DEFAULT NOW() | Дата создания записи |

Для суммы используется тип `NUMERIC(10,2)`, а не `FLOAT` — это важно, потому что числа с плавающей точкой дают погрешности при арифметике, что недопустимо для финансовых данных. Ограничение `CHECK (amount > 0)` гарантирует, что нельзя добавить платёж с нулевой или отрицательной суммой.

## 5. Перечисления (ENUM)

В базе используется 6 перечислений. Они позволяют хранить только допустимые значения и избавляют от ошибок вроде «оплачен» вместо «оплачено».

| Тип | Значения |
|-----|----------|
| gender_type | М, Ж |
| appointment_type | консультация, анализ, процедура |
| appointment_status | запланирован, проведен, отменен |
| payment_type | оплата приёма, оплата процедуры |
| payment_method | наличные, карта, онлайн |
| payment_status | оплачено, не оплачено |
| user_role | администратор, врач, регистратор |

## 6. Связи между таблицами

| Связь | Тип | Правило удаления |
|-------|-----|-----------------|
| patients → appointments | 1:M | CASCADE — удаление пациента удаляет все его приёмы |
| doctors → appointments | 1:M | CASCADE |
| patients → medical_records | 1:M | CASCADE |
| doctors → medical_records | 1:M | **RESTRICT** — нельзя удалить врача с медкартами |
| medical_records → lab_tests | 1:M | CASCADE — вместе с картой удаляются анализы |
| patients → payments | 1:M | CASCADE |
| doctors → users | 1:1 (опц.) | SET NULL — удаление врача не удаляет учётную запись |

Разные политики удаления выбраны осознанно. Например, CASCADE для приёмов логичен — если пациент исключён из системы, его расписание тоже не нужно. А RESTRICT для медкарт — это защита медицинских данных от случайного уничтожения.

## 7. Индексы

Для ускорения часто выполняемых запросов созданы 8 индексов:

```sql
-- Поиск всех приёмов пациента / врача / по дате
CREATE INDEX idx_appointments_patient  ON appointments (patient_id);
CREATE INDEX idx_appointments_doctor   ON appointments (doctor_id);
CREATE INDEX idx_appointments_date     ON appointments (appointment_date);

-- Медицинские карты по пациенту и врачу
CREATE INDEX idx_medical_records_patient ON medical_records (patient_id);
CREATE INDEX idx_medical_records_doctor  ON medical_records (doctor_id);

-- Анализы по медкарте
CREATE INDEX idx_lab_tests_record ON lab_tests (medical_record_id);

-- Платежи по пациенту и дате
CREATE INDEX idx_payments_patient ON payments (patient_id);
CREATE INDEX idx_payments_date    ON payments (payment_date);
```

Без индексов при большом объёме данных каждый запрос «история пациента» или «расписание на дату» выполнял бы полное сканирование таблицы. С индексами PostgreSQL использует B-tree поиск, что на порядки быстрее.

## 8. Представления

Созданы два представления для удобства работы с данными.

### `v_appointments` — Приёмы с именами

```sql
CREATE VIEW v_appointments AS
SELECT
    a.id,
    a.appointment_date,
    a.appointment_time,
    a.type,
    a.status,
    p.last_name || ' ' || p.first_name AS patient_name,
    d.last_name || ' ' || d.first_name AS doctor_name,
    d.specialty,
    a.notes
FROM appointments a
JOIN patients p ON p.id = a.patient_id
JOIN doctors  d ON d.id = a.doctor_id;
```

Это представление избавляет от необходимости каждый раз писать JOIN при выводе расписания. Вместо громоздкого запроса достаточно написать:

```sql
SELECT * FROM v_appointments WHERE appointment_date = '2026-05-06';
```

### `v_patient_summary` — Сводка по пациентам

```sql
CREATE VIEW v_patient_summary AS
SELECT
    p.id,
    p.last_name || ' ' || p.first_name AS full_name,
    p.birth_date,
    p.phone,
    COUNT(DISTINCT a.id)  AS total_appointments,
    COUNT(DISTINCT mr.id) AS total_records,
    COALESCE(SUM(py.amount) FILTER (WHERE py.status = 'оплачено'), 0) AS total_paid
FROM patients p
LEFT JOIN appointments    a  ON a.patient_id  = p.id
LEFT JOIN medical_records mr ON mr.patient_id = p.id
LEFT JOIN payments        py ON py.patient_id = p.id
GROUP BY p.id, p.last_name, p.first_name, p.birth_date, p.phone;
```

Показывает по каждому пациенту: количество приёмов, количество записей в медкарте и сумму фактически оплаченных услуг. Удобно для административного раздела системы.

## 9. Примеры запросов

В файле `queries.sql` реализовано 10 типовых запросов. Ниже описан каждый из них.

### Запрос 1. Расписание на конкретный день

Выводит всех пациентов и врачей, принимающих в указанный день, упорядоченных по времени.

```sql
SELECT
    a.appointment_time        AS время,
    p.last_name || ' ' || p.first_name AS пациент,
    d.last_name || ' ' || d.first_name AS врач,
    d.specialty               AS специальность,
    a.type                    AS тип_приёма,
    a.status                  AS статус
FROM appointments a
JOIN patients p ON p.id = a.patient_id
JOIN doctors  d ON d.id = a.doctor_id
WHERE a.appointment_date = '2026-04-10'
ORDER BY a.appointment_time;
```

Используется на стойке регистратуры, чтобы видеть кто и когда сегодня принимает.

### Запрос 2. История приёмов пациента

Вся история визитов конкретного пациента от последнего к первому.

```sql
SELECT
    a.appointment_date, a.appointment_time,
    d.last_name || ' ' || d.first_name AS врач,
    d.specialty, a.type, a.status, a.notes
FROM appointments a
JOIN doctors d ON d.id = a.doctor_id
WHERE a.patient_id = 1
ORDER BY a.appointment_date DESC, a.appointment_time;
```

### Запрос 3. Медицинская карта со всеми анализами

Показывает все записи из медкарты пациента вместе с результатами лабораторных исследований через LEFT JOIN — чтобы показать записи даже без анализов.

```sql
SELECT
    mr.visit_date, d.last_name || ' ' || d.first_name AS врач,
    mr.diagnosis, mr.prescription, mr.notes,
    lt.test_name, lt.test_date, lt.result
FROM medical_records mr
JOIN doctors d ON d.id = mr.doctor_id
LEFT JOIN lab_tests lt ON lt.medical_record_id = mr.id
WHERE mr.patient_id = 2
ORDER BY mr.visit_date DESC, lt.test_date;
```

### Запрос 4. Загруженность врачей за период

Считает количество проведённых приёмов по каждому врачу за указанный месяц. Отменённые приёмы исключаются через `status != 'отменен'`.

```sql
SELECT
    d.last_name || ' ' || d.first_name AS врач,
    d.specialty, COUNT(a.id) AS кол_во_приёмов
FROM doctors d
LEFT JOIN appointments a ON a.doctor_id = d.id
    AND a.appointment_date BETWEEN '2026-04-01' AND '2026-04-30'
    AND a.status != 'отменен'
GROUP BY d.id, d.last_name, d.first_name, d.specialty
ORDER BY кол_во_приёмов DESC;
```

### Запрос 5–6. Финансовые отчёты

Детальный список платежей за период и сводный итог — сколько оплачено, сколько нет. Нужны бухгалтерии и администрации.

### Запрос 7. Предстоящие запланированные приёмы

Все пациенты с предстоящими записями — используется для напоминаний и подтверждения визитов.

### Запрос 8. Неактивные пациенты

Находит пациентов, которые не приходили более 90 дней. Полезно для маркетинговых рассылок или анализа оттока.

```sql
HAVING MAX(a.appointment_date) < CURRENT_DATE - INTERVAL '90 days'
    OR MAX(a.appointment_date) IS NULL
```

`IS NULL` нужен для пациентов, у которых вообще не было проведённых приёмов.

### Запрос 9. Расписание врача на неделю вперёд

Личное расписание конкретного врача на ближайшие 7 дней с контактами пациентов.

### Запрос 10. Поиск пациента по фамилии

```sql
WHERE last_name ILIKE '%ивано%'
```

`ILIKE` — регистронезависимый поиск в PostgreSQL. Поиск по частичному совпадению удобен, если регистратор не знает точную фамилию.

| Таблица | Количество записей |
|---------|-------------------|
| patients | 8 |
| doctors | 6 |
| users | 6 |
| appointments | 12 |
| medical_records | 8 |
| lab_tests | 8 |
| payments | 11 |

Врачи представляют 6 разных специальностей: терапевт, кардиолог, хирург, невролог, офтальмолог, эндокринолог.

Среди приёмов есть все три статуса:
- **проведенные** (8 штук) — за апрель 2026 г.;
- **запланированные** (3 штуки) — на май 2026 г.;
- **отменённый** (1 штука) — пациент не явился.

Медицинские карты содержат реальные диагнозы и назначения: ВСД, гипертензия, постоперационный период, ОРВИ, мигрень, экстрасистолия, миопия, сахарный диабет 2 типа.

Среди платежей есть как оплаченные (8), так и неоплаченные (3 — для запланированных приёмов).

Тестовые учётные записи используют одинаковый пароль `password123`, захэшированный через bcrypt. В реальной системе каждый пользователь устанавливает свой пароль при первом входе.

## 11. Обеспечение целостности данных

В базе реализовано несколько уровней защиты данных:

**Ограничения на уровне столбцов:**
- `NOT NULL` — для обязательных полей;
- `UNIQUE` — для телефонов, email, логинов;
- `CHECK (amount > 0)` — сумма платежа не может быть нулевой или отрицательной;
- `DEFAULT` — автоматические значения для дат и статусов.

**Внешние ключи:**
- Гарантируют, что нельзя создать приём для несуществующего пациента или врача;
- Политики `CASCADE` / `RESTRICT` / `SET NULL` контролируют поведение при удалении.

**ENUM-типы:**
- Исключают ввод произвольных строк в поля статусов и ролей.

**Уникальное ограничение на расписание:**
- `UNIQUE (doctor_id, appointment_date, appointment_time)` предотвращает двойную запись к врачу.
